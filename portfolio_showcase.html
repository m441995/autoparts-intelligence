Production-Grade Python

safety_stock.py def calculate_safety_stock( demand_std_daily: float, avg_lead_time: float, lead_time_std: float, avg_daily_demand: float, z: float, ) -> int: """ Combined demand + lead time variability safety stock. SS = Z × √( LT × σ_d² + d̄² × σ_LT² ) Superior to simple Z×σ×√LT — accounts for supplier variability. """ demand_variance = avg_lead_time * (demand_std_daily ** 2) leadtime_variance = (avg_daily_demand ** 2) * (lead_time_std ** 2) combined_std = np.sqrt(demand_variance + leadtime_variance) return max(0, int(np.ceil(z * combined_std)))
abc_xyz_classifier.py def classify_abc( revenue_series: pd.Series, a_threshold: float = 80.0, b_threshold: float = 95.0, ) -> pd.Series: """Pure-function ABC classification. Vectorized — no row loops.""" total = revenue_series.sum() sorted_rev = revenue_series.sort_values(ascending=False) cum_pct = sorted_rev.cumsum() / total * 100 abc = pd.cut( cum_pct, bins=[0, a_threshold, b_threshold, 100], labels=["A", "B", "C"], ) return abc.reindex(revenue_series.index)

Interview Preparation

Key Talking Points

Prepared answers for the questions hiring managers always ask.

Q: Why did you use SARIMA over Prophet?
The pipeline uses both. SARIMA is chosen first for 24+ month series — it gives interpretable seasonal parameters and handles automotive demand patterns well. Prophet is available as an ensemble addition. If SARIMA MAPE exceeds 30% on holdout, the pipeline automatically falls back to ETS. Model selection is data-driven, not arbitrary.
Q: How did you handle class imbalance in the ML model?
Stockouts are typically 3–8% of inventory positions — a significant class imbalance. Random Forest uses class_weight='balanced' which auto-adjusts sample weights inversely proportional to class frequency. XGBoost uses scale_pos_weight = n_negative / n_positive. I also evaluate on Precision-Recall AUC, not just accuracy, since accuracy is misleading on imbalanced datasets.
Q: What's different about your safety stock formula?
Most systems use Z × σ_d × √LT — which assumes constant lead time. In automotive supply chains with overseas suppliers (Japan, Germany), lead time variance is significant. The combined formula SS = Z × √(LT × σ_d² + d̄² × σ_LT²) captures both sources of variability. For a part with high demand stability but highly variable lead time, this makes a major difference in buffer sizing.
Q: How would you deploy this in production?
The pipeline is CLI-driven and stage-independent — each stage can run standalone. For production: Airflow DAGs would orchestrate the stages nightly, with results written to PostgreSQL (partitioned by snapshot_date for query performance). Power BI connects via DirectQuery or scheduled import. The model artifacts are joblib-serialized and versioned. Monitoring via loguru logs shipped to ELK stack.
Q: How does this align with what you did at the dealership?
Every formula and classification in this platform comes from real operational problems I faced. The dead stock detection thresholds, the ABC cutoffs, the lead time variability — these mirror what I saw in practice. The SAP table references (MARA, MARD, VBAP, EKPO) are the exact tables I extracted from. This isn't a textbook exercise — it's a formalized version of decisions I made manually for years.
Q: Why PostgreSQL over a cloud data warehouse?
PostgreSQL was chosen for accessibility and to demonstrate schema design fundamentals — anyone can run it locally. The architecture is cloud-agnostic: the same schema works on Amazon RDS, Azure Database for PostgreSQL, or Google Cloud SQL. For a production scaled system, I'd move heavy aggregations to dbt + Snowflake or BigQuery, but the analytical logic stays identical. The Parquet output format was specifically chosen to make that migration trivial.
